# Trading Bot v4 — Phase 2 Design (Risk Kernel + Populated Locks + Kill Switches)

**Source plan:** Plan v4 §4 (validation policy lock), §6 (Risk kernel — numerical limits + kill switches), §10 (failure mode catalog), §14 P0 items (`Strategy identity in every risk decision`, `Crypto exposure cap enforced`, `PDT — block entries only, never exits`, `Hash chain verified at startup and nightly`, `Single-writer guard`, `Role-persona hash check (L3/L8)`, `Adversarial pair on every hypothesis` — partial; full check Phase 5).

**Phase duration in plan:** 2 calendar weeks. **Trading remains halted.**

## Goal

Ship the **L6 risk kernel**: the final veto layer between strategy intents and execution. Every intent passes through `risk.precheck.evaluate()` before reaching `execution/` (Phase 1 + Phase 3 wire that path). Risk decisions are logged via `ledger.write_decision`. Kill switches are detectors that, when they trip, flip the system into "halt new entries" and the operator must intervene.

Phase 2 also **populates** the nine `.lock` files with the real numerics from Plan §4 / §6 / §9, and ships a boot-time hash check that refuses to start the system if any lock file's content differs from `policy/HASHES`.

## Limit categories (Plan §6, exact numerics)

| Tier | Limit | Seed value | On breach |
|---|---|---|---|
| Account | Daily realized + unrealized DD | 1.0% of equity at session start | halt new entries; existing positions managed by exits only |
| Account | Trailing peak-to-trough DD | 5.0% over 60-day window | lane mode → observe-only; new validation lock required to resume |
| Account | PDT (entry-side, never blocks exits) | ≤3 day trades / rolling 5 BD when equity < $25k | block entries whose same-day exit would be the 4th day-trade; emergency exits always allowed |
| Asset class | Equity gross | ≤80% of equity | reduce-only mode; new buys rejected |
| Asset class | Crypto gross | ≤15% of equity | reduce-only mode |
| Asset class | Options BP utilisation | ≤30% of equity | no new option opens |
| Lane | Per-lane allocation | ≤40% of equity | lane status → reduce-only |
| Lane | Per-lane daily loss | ≤0.5% of equity | lane halted for the session |
| Strategy | Per-strategy realized + unrealized loss 30d | ≤1.5% of equity | strategy moved to observe-only; mutation request opened |
| Symbol | Per-symbol gross | ≤5% of equity | order qty reduced to fit; reason logged |
| Order | Per-order at-risk capital | ≤2% of equity | order rejected; reason logged |
| Order | Stop coverage required | every open position must have an exit plan within 1 min of fill | missing stop after 1 min → auto-flatten the position (lands in Phase 3 when execution sequencer ships) |

The strictest rule wins. All checks evaluate before the order leaves the kernel; failure routes to `risk_decision.reason` and the row is logged in `strategy_decision`.

## Kill switches (Plan §6 + §10)

Any one trips → all new entries halt; existing positions can only be reduced.

1. **Reconciliation mismatch** — `bot_hash != broker_hash` at any window.
2. **Unknown classification** — any open position with `classification='unknown'` older than 15 minutes.
3. **Data freshness breach** — bar/quote staler than per-lane threshold (equity 5 min, crypto 60 s, options 60 s during RTH).
4. **Policy hash mismatch** — any `.lock` modified without `HASHES` regen.
5. **Broker API error rate** > 5% over rolling 5 minutes (wires in Phase 3).
6. **Clock skew** — wall-clock vs server-clock > 2 s.
7. **SQLite integrity_check** failed at startup.
8. **Intraday PnL floor** — daily realized PnL below −1.5% of session-start equity.

Phase 2 ships the detector library + the kill-state table + the halt router. The runtime drivers (live broker error rate, live data freshness watermarks, live clock skew) wire in Phases 3 / 5 when the corresponding adapters / daemon ship; Phase 2 includes the helper functions and unit tests that drive each detector from a fixture.

## Single-entry contract

```python
risk.precheck.evaluate(
    conn,                   # ledger writer connection (read-only OK for evaluation;
                            # writes happen via write_decision)
    intent: OrderIntent,
    account: AccountState,  # equity, equity_at_session_start, day_trade_count
    positions: Sequence[Position],   # current bot+broker positions
    policy: PolicyBundle,   # loaded locks
    kill_state: KillSwitchState,
    now: datetime,
) -> RiskDecision
```

`RiskDecision` is a frozen dataclass: `verdict ∈ {accept, reduce, halt}`, `reason` (string), `adjusted_qty` (Optional[float]).

The caller (Phase 3 execution router) translates `verdict` into the next action:

- `accept` → submit the order at `intent.qty`.
- `reduce` → submit at `adjusted_qty`.
- `halt` → do not submit; `write_decision(..., risk_decision='halt', risk_reason=...)`.

The `strategy_decision` row is always written, regardless of verdict. This satisfies Plan §14 P0: "Strategy identity in every risk decision".

## Kill-state table (in `ledger.db`)

```sql
CREATE TABLE kill_switch_state (
    ledger_seq    INTEGER PRIMARY KEY AUTOINCREMENT,
    detector      TEXT NOT NULL,                -- one of the 8 kill types
    fired_at      TEXT NOT NULL,
    fired_reason  TEXT NOT NULL,
    cleared_at    TEXT,                         -- NULL while active
    cleared_by    TEXT,                         -- operator notation
    prev_hash     TEXT NOT NULL,
    this_hash     TEXT NOT NULL
);
```

Hash-chained like the rest of the ledger. Append-only (UPDATE forbidden); "clear" is recorded as a new row with the same detector + `cleared_at` set (NULL `fired_at` to distinguish).

For Phase 2 simplicity, "clear" is a separate row type `cleared` event:

```sql
CREATE TABLE kill_switch_event (
    ledger_seq    INTEGER PRIMARY KEY AUTOINCREMENT,
    event_ts      TEXT NOT NULL,
    detector      TEXT NOT NULL,
    event_kind    TEXT NOT NULL,          -- fire | clear
    reason        TEXT,
    actor         TEXT NOT NULL,          -- operator | system
    prev_hash     TEXT NOT NULL,
    this_hash     TEXT NOT NULL
);
```

A detector is "active" iff its latest event is `fire`. A view `kill_switch_active` derives the active set.

## Boot path

`kernel/boot.py::run_boot_checks(ledger_db, mirror_db, policy_dir)` runs in order:

1. `sqlite3 PRAGMA integrity_check` on both DBs.
2. `ledger.read_schema_version(conn) == SCHEMA_VERSION`.
3. Walk `policy/HASHES`; each listed file's SHA-256 must match.
4. `ledger.verify_all_chained(conn)` for both ledger and mirror.
5. `risk.kill_switches.active_kills(conn)` — if non-empty, the kernel may start in "halt new entries" mode but never in "permit entries" mode.

Failure at step 1–4 → refuse to start (exit non-zero from `tools/boot_check.py`). Step 5 is informational.

## P0 / P1 acceptance items satisfied

P0:

- Three policy lock files + HASHES — **populated** (validation, risk, pdt; the other six remain skeletons where the plan defers them).
- Strategy identity in every risk decision — `risk.precheck.evaluate` writes `strategy_decision` with `strategy_id` set.
- Crypto exposure cap enforced — `asset_class_caps.evaluate` returns `reject` with `reason='asset_class_cap:crypto'` when over 15%.
- PDT — block entries only, never exits — `pdt.evaluate` blocks entries; emergency-exit origin always passes; `day_trade_count` read from the broker.
- Hash chain verified at startup — `kernel/boot.py` runs `verify_all_chained` on every boot.
- Single-writer guard — already shipped Phase 1; `kernel/boot.py` asserts the writer lock before any write.

P1:

- Asset-class concentration check — replaces v2 factor-crowding test; `asset_class_caps` enforces.
- Validation lock cooldown — `policy_loader.honor_cooldown(lock, prior_lock)` returns `True` only when 7 days have elapsed since the loosened version was committed; tightening is immediate.
- Autonomous demotion — `lane_caps.demote_on_breach` transitions a lane to `observe-only` and writes the transition into the lane state table.

- Live data freshness watermarks (needs L1 ingest writers).

## Risks

- **Risk:** Tests run against fixture data; do not exercise the live broker. **Mitigation:** Phase 3 will add live-data integration tests once the Alpaca adapter is hardened.
- **Risk:** The 9 lock files are now content-stable; HASHES regeneration becomes part of every lock change. **Mitigation:** `tools/recompute_hashes.py --check` runs in CI / pre-commit / boot; the boot check refuses to start on mismatch.
